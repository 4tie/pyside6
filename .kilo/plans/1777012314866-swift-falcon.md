# Plan: Replace WebSocket with Polling for Backtest Status Updates

## Problem
The WebSocket connection to `ws://127.0.0.1:8000/ws/backtest` is failing, causing errors in the console and degrading user experience. The application already has a polling fallback mechanism, but it's only triggered when WebSocket fails.

## Solution
Remove WebSocket implementation entirely and rely solely on HTTP polling for status updates. This simplifies the code and eliminates connection failures.

## Changes Required

### 1. Remove WebSocket Variables and Functions
- Delete `ws` and `wsConnected` variables (lines 13-14)
- Delete `connectWebSocket()` function (lines 19-51)
- Delete `handleWebSocketMessage()` function (lines 53-70)
- Delete WebSocket initialization call on page load (line 115)

### 2. Modify Backtest Execution Flow
- In the run backtest button click handler (lines 294-363):
  - After successful POST to `/api/backtest/execute`, call `startPolling()` instead of relying on WebSocket
  - Keep existing error handling for the initial request

### 3. Maintain Polling Logic
- Keep existing `startPolling()` function (lines 72-103) unchanged
- Keep existing `resetBacktestButton()` function (lines 105-112) unchanged
- Keep existing polling interval and currentRunId variables (lines 15-16)

### 4. Cleanup
- Remove WebSocket-related comments
- Ensure no references to WebSocket remain in the code

## Implementation Details
- The polling mechanism already exists and handles:
  - Status checks every 2 seconds
  - Completion detection (redirects to run detail page)
  - Error handling (resets button state)
  - Automatic start/stop of polling intervals
- No changes needed to backend API since polling uses existing `/api/backtest/status` endpoint
